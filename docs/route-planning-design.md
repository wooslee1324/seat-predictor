# 경로 기반 좌석 예측 설계 (출시형 구조)

지금 데모는 "출발지 → 도착지" 텍스트만 입력받아 하나의 곡선을 보여줍니다. 출시형
서비스가 되려면 **실제 탈 수 있는 지하철 호선·버스 번호로 현실적인 경로 후보를
만들고, 그 경로마다 좌석 확률을 예측**해야 합니다. 이 문서는 그 구조를 **설계만**
합니다(이번 단계에서 전체 구현은 하지 않습니다).

## 0. 원칙

- **현재 데모를 망가뜨리지 않는다.** 기존 `app.py`·`mobile_demo.py`·`core`·
  `backend` 는 그대로 두고, 경로 계층을 **추가**하는 방향으로 확장한다.
- **정보를 섞지 않는다.** 네 가지를 화면·응답에서 항상 구분한다.
  - `실시간 도착정보` (API가 주는 사실)
  - `예상 혼잡도` (통계 또는 실시간 혼잡 신호에서 추정)
  - `예상 좌석 확률` (혼잡도로 유도한 추정치, 실측 아님)
  - `참고용 통계` (과거 집계)
- **API가 안 주는 값을 실시간이라 부르지 않는다.** 특히 좌석 수는 어떤 API도
  실시간으로 주지 않으므로 항상 "예상"으로만 표기한다.

## 1. 출시형 데이터 흐름 개요

```
[출발지/현재위치] ──지오코딩──▶ (위도·경도)
        │
        ├─▶ 근접 지하철역 + 호선 조회
        └─▶ 근접 버스정류장 + 경유노선 조회
                    │
        [출발/도착 좌표] ──경로탐색(ODsay 등)──▶ 경로 후보 목록
                    │                            (지하철/버스/환승 조합)
                    ▼
        각 경로 구간마다:
          · 실시간 도착정보(지하철/버스)      ← 사실
          · 실시간 혼잡/만차(버스 제공 시)     ← 참고 실시간 신호
          · 시간대별 혼잡도 통계(subwConfusion) ← 통계
          · 승하차 통계                        ← 참고용 통계
                    ▼
          core.prediction → 예상 좌석 확률(추정) + 데이터 출처 배지
                    ▼
        [경로별 카드]: 호선/버스번호 · 역/정류장 · 도착예정 · 혼잡 · 예상 좌석 확률
```

## 2. 필요한 데이터 · API 목록

| 용도 | API / 데이터 | 포털·인증키 | 성격 | 비고 |
| --- | --- | --- | --- | --- |
| 주소·장소 → 좌표(지오코딩) | 카카오/네이버 로컬 API, 또는 ODsay POI 검색 | 각 사 키(민간) | 조회 | "현재 위치"는 폰 GPS로 대체 가능 |
| 근접 지하철역 + 좌표 | 서울시 지하철역 정보(좌표) 데이터셋 | 서울 열린데이터광장 | 통계(정적) | 역-호선 매핑 포함 |
| 역 → 호선 매핑 | 위 데이터셋 / `subwConfusion` 의 노선 필드 | 열린데이터광장 | 통계(정적) | 이미 일부 확보 |
| 근접 버스정류장 | 좌표기반 근접 정류소 조회 `getStationByPos` | 공유자원포털/공공데이터포털 | 조회 | `tmX,tmY,radius` |
| 정류장 → 경유 버스번호 | 정류소별 경유노선목록 `getRouteByStation` | 공공데이터포털 | 조회 | 정류소 지나는 노선 |
| 경로 후보(출발→도착) | ODsay 대중교통 길찾기 API | ODsay 키(민간) | 조회(계산) | 지하철·버스·환승 통합 경로. 좌표(SX,SY,EX,EY) 필요 |
| 실시간 지하철 도착 | `realtimeStationArrival` | 열린데이터광장 | **실시간** | 이미 연동(`/realtime/subway/arrivals`) |
| 실시간 버스 도착 + 혼잡/만차 | `getStationByUid` | TOPIS/공공데이터포털 | **실시간** | 이미 연동(`/realtime/bus/arrivals`) |
| 시간대별 혼잡도 | `subwConfusion` | 열린데이터광장 | **통계**(분기) | 이미 연동 |
| 역/정류장 승하차 | `CardSubwayStatsNew`·`CardBusStatisticsServiceNew` | 열린데이터광장 | **통계**(일, 3일 지연) | 참고용 통계 |
| (선택) 칸별 실시간 혼잡도 | TMAP/SK 지오비전 퍼즐 | 민간 키·비용 | 준실시간 | 정확도↑, 비용·약관 확인 |

> 경로탐색(ODsay)과 근접 정류소/지오코딩은 **새로 필요한 축**입니다. 실시간·통계
> API는 이미 프로젝트에 있는 것을 재사용합니다.

## 3. 좌석 확률 계산에 넣을 정보 조합(설계)

좌석 확률은 여전히 `core.prediction` 이 계산하되, 입력 "혼잡도"를 아래 우선순위로
가장 실시간에 가까운 값으로 채택하고 **출처를 항상 함께 반환**합니다.

1. 실시간 혼잡 신호(버스 혼잡/만차, 또는 칸별 실시간 혼잡) → 출처 `실시간`
2. 없으면 시간대별 혼잡도 통계(`subwConfusion`) → 출처 `예상(시간대 평균)`
3. 그것도 없으면 러시아워 곡선 mock → 출처 `추정(Mock)`

보조 입력(가중치는 추후 튜닝):

- **도착 예정 시간 / 대기시간**: 실시간 도착정보로 "N분 뒤 탑승" 시나리오를 실제
  값으로 대체. 지연이 크면 "예측 신뢰도 낮음" 표시.
- **승하차 통계**: 해당 역/정류장이 원래 붐비는 곳인지 보정(참고용).
- **버스 만차 여부**: 만차면 좌석 확률을 크게 낮춤(단, 이는 "좌석 수"가 아니라
  혼잡 신호로만 반영).

핵심: 이 조합의 결과는 언제나 **예상 좌석 확률(추정치)** 이며, 실제 좌석 수가
아닙니다.

## 4. 백엔드 endpoint 설계 초안 (향후)

기존 `/health`·`/stations`·`/predict`·`/realtime/*` 위에 **경로 계층**을 추가합니다.

- `GET /nearby?lat={위도}&lon={경도}&radius={m}`
  → 주변 이용 가능한 교통수단. 응답: 가까운 지하철역(+호선), 가까운 버스정류장
    (+경유 버스번호). 출처: 근접 정류소/지하철역 데이터.
- `GET /routes?from_lat=&from_lon=&to_lat=&to_lon=` (또는 `from=역/주소&to=…`)
  → 실제 가능한 **경로 후보 목록**. 각 후보는 구간(지하철 호선/버스 번호,
    승하차 역·정류장, 예상 소요시간, 환승 정보)으로 구성. 출처: 경로탐색(ODsay).
- `POST /routes/predict`
  → 요청: 경로 후보(구간 목록). 응답: 각 구간·경로에 대해 실시간 도착정보 +
    예상 혼잡도 + **예상 좌석 확률(추정)** + 각 값의 출처. 내부적으로 기존
    `/realtime/*` 와 `core.prediction` 을 재사용.

응답 스키마 원칙: 구간마다 `line`/`route_name`, `from_stop`/`to_stop`,
`arrival`(실시간, 있을 때), `congestion`(출처 라벨 포함), `seat_probability`
(항상 "추정"), `data_sources`(각 값의 출처) 를 분리해서 담습니다.

## 5. 모바일 화면 흐름 초안

1. **출발/도착 입력** — 현재 위치(GPS) 또는 출발지 검색 + 도착지 검색.
2. **주변 교통수단(선택)** — "가까운 역 2호선 강남역, 버스 146·401 …" 미리보기.
3. **경로 후보 목록** — 카드로 표시:
   - `2호선 강남역 → 사당역` · 도착 3분 · 예상 혼잡 높음 · 예상 좌석 12%
   - `146번 강남역10번출구 → 사당역4번출구` · 도착 5분 · 만차 신호 · 예상 좌석 8%
   - `2호선 + 마을버스 환승` · … 
   각 카드에 **실시간/예상/통계 배지**를 분명히 표시.
4. **경로 상세** — 선택한 경로의 시간대별 좌석 확률 곡선, 추천 대기 위치(칸/문),
   "지금 vs N분 뒤" 추천. 좌석은 항상 "예상"으로 표기.
5. **저장/알림(선택)** — 자주 쓰는 경로 저장, 출발 시간 알림.

## 6. 데모 단계 우선 최소 기능(MVP) 제안

현재 데모를 깨지 않으면서 "경로 느낌"을 먼저 보여줄 수 있는 최소 조합입니다.

1. **역-호선 자동 표시** — 출발/도착 지하철역을 고르면 해당 **호선**을 함께 표시
   (이미 있는 `subwConfusion` 노선 정보 활용). 경로탐색 없이도 "2호선 강남역 →
   사당역"처럼 현실적으로 보이게.
2. **정류장 경유 버스번호 표시** — 버스 모드에서 정류장의 경유 노선(버스번호)을
   보여줌(`getRouteByStation`). 좌석 예측은 기존 로직 재사용.
3. **경로 후보는 "단일 수단"부터** — 지하철 단독 / 버스 단독 후보만 먼저. **환승
   경로(ODsay)** 는 그다음 단계로 미룸(외부 API·비용 때문).
4. **실시간 도착정보 배지 연결** — 이미 만든 `/realtime/*` 를 붙여 "도착 N분"만
   추가 표시(좌석 확률과 분리).

즉 MVP는 "**경로탐색(ODsay) 없이, 역-호선/정류장-버스번호 + 기존 예측 + 실시간
도착정보**"로 현실감을 크게 올리고, 진짜 환승 경로탐색은 정식 출시 단계로 넘깁니다.

## 7. 단계적 로드맵

1. (MVP) 역→호선, 정류장→버스번호 표시 + 실시간 도착정보 배지. 데모 확장, 신규
   외부 API 없음.
2. 근접 교통수단(`getStationByPos` 등)으로 "주변에서 탈 수 있는 것" 화면 추가.
3. 경로탐색(ODsay) 도입 → 진짜 경로 후보(환승 포함) + `/routes`·`/routes/predict`.
4. 좌석 확률 계산에 실시간 혼잡/만차·지연 가중치 반영, 출처 배지 정교화.
5. (선택) 칸별 실시간 혼잡도(TMAP/SK)로 정확도·추천 대기 위치 강화.

## 8. 인증키·비용 주의

- 신규로 **민간 API 키**(경로탐색 ODsay, 지오코딩 카카오/네이버)가 필요해질 수
  있음 → 호출 한도·비용·약관 사전 확인. 서울 공공데이터(무료)로 가능한 범위를
  먼저 최대한 활용.
- 모든 인증키는 환경변수/`secrets.toml` 로 관리하고 **절대 커밋하지 않음**.
- 실시간 API는 짧은 캐시, 통계·정적 데이터(역 좌표 등)는 긴 캐시로 분리.
