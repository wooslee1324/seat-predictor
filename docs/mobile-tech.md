# 모바일 발전 방향과 기술 추천 (PWA 우선)

이 문서는 좌석 예측기를 모바일로 발전시킬 때의 기술 선택지를 비교하고, 우리
프로젝트에 맞는 방향을 추천합니다. 결론부터 말하면 **PWA(모바일 웹앱)를 1차
목표로 추천**합니다.

## 선택지 비교

| 방향 | 장점 | 단점 | 우리 상황 적합도 |
| --- | --- | --- | --- |
| PWA (모바일 웹앱) | 배포가 가장 빠름, 앱스토어 심사 없음, 웹 자산 재사용, 설치 없이 URL로 접근, 홈 화면 추가·오프라인 캐싱·푸시 일부 지원 | 아이폰에서 일부 네이티브 기능 제약, 완전한 네이티브 감성은 아님 | ★★★ 매우 높음 |
| React Native | 하나의 코드로 iOS·Android 네이티브 앱, 앱스토어 배포, 네이티브 기능 풍부 | 앱스토어 심사·계정·빌드 환경 필요, 학습·운영 비용 큼 | ★★ 중간(향후 선택) |
| Flutter | 성능 좋은 네이티브 앱, 예쁜 UI 구성 쉬움 | 팀이 Dart 언어를 새로 배워야 함, 웹 자산 재사용 어려움 | ★ 낮음(현 시점) |

## 왜 PWA를 먼저 추천하는가

첫째, **가장 빠르게 실제 사용자에게 닿을 수 있습니다.** 앱스토어 심사나 개발자
계정 없이 URL만으로 배포되고, 사용자는 브라우저에서 "홈 화면에 추가"로 앱처럼
쓸 수 있습니다. 수업 발표에서 실제 서비스로 넘어가는 단계에 가장 마찰이 적습니다.

둘째, **지금까지의 자산을 그대로 살립니다.** `mobile_demo.py`에서 이미 스마트폰
화면 흐름(경로 입력 → 좌석 확률 → 추천 위치 → 그래프 → 경로 저장)을 검증했습니다.
이 흐름을 그대로 웹 프런트로 옮기면 되므로 기획을 다시 할 필요가 없습니다.

셋째, **백엔드를 한 번만 만들면 재사용됩니다.** `core/`를 감싸는 REST API
([deployment-plan.md](deployment-plan.md) 3단계)를 만들어 두면, PWA가 그 API를
호출합니다. 나중에 네이티브 앱(React Native)이 필요해져도 같은 API를 그대로 쓸 수
있어 이 투자는 버려지지 않습니다.

## 추천 기술 스택 (PWA)

- 프레임워크: **React + Vite** (또는 팀이 익숙하면 Next.js). 가볍고 학습 자료가
  많으며 PWA 설정이 간단합니다.
- PWA 구성: **Vite PWA 플러그인**으로 서비스 워커·매니페스트·오프라인 캐싱을
  손쉽게 추가합니다.
- 차트: **Recharts** 또는 **Chart.js** — 지금 Plotly로 그리는 시간대별 좌석 확률
  그래프를 대체합니다.
- 지도(향후): 역·정류장 위치를 보여주고 싶다면 **Leaflet**(오픈소스) 또는 카카오
  지도 API를 검토합니다.
- 백엔드 연동: `fetch`로 우리 백엔드 API(JSON) 호출. 서울 공공데이터 인증키는
  백엔드에만 두고 프런트에는 노출하지 않습니다.

## 단계별 진행 제안

1. 백엔드 API 최소 버전을 먼저 만듭니다(`core.build_prediction`을 REST로 노출).
2. React + Vite로 PWA 프로젝트를 만들고, `mobile_demo.py`의 화면 흐름을 옮깁니다.
3. 화면이 API에서 데이터를 받아오도록 연결합니다.
4. Vite PWA 플러그인으로 "홈 화면 추가"·오프라인 캐싱을 붙입니다.
5. 정적 호스팅(Vercel 등)에 배포하고 실제 스마트폰에서 테스트합니다.

## 나중에 네이티브 앱이 필요해진다면

푸시 알림, 위치 기반 백그라운드 동작 등 네이티브 기능이 꼭 필요해지는 시점이
오면 그때 **React Native**를 검토합니다. PWA를 React로 만들어 두면 컴포넌트 설계
개념과 API 연동을 상당 부분 재활용할 수 있어, 네이티브로의 전환 비용이 낮아집니다.
지금 단계에서 네이티브부터 시작하는 것은 배포·운영 부담이 커서 권장하지 않습니다.
