# 배포 계획과 데모 · 프로덕션 분리

이 문서는 지금의 Streamlit 프로토타입을 실제 배포 가능한 서비스로 키워 가기 위한
단계별 계획과, 발표용 데모 코드와 실제 서비스용 코드를 어떻게 나눌지에 대한
제안을 담습니다.

## 현재 상태 진단

지금 프로젝트는 하나의 저장소 안에서 화면 코드와 로직 코드가 섞여 있는 단일
Streamlit 앱이었습니다. 프로토타입으로는 훌륭하지만, 실제 서비스로 확장할 때 다음
한계가 있었습니다.

첫째, 예측 로직이 `app.py`와 `mobile_demo.py`에 복제되어 있어 한쪽을 고치면
다른 쪽과 어긋날 위험이 있었습니다. 둘째, 데이터 연동 코드가 Streamlit의 캐시에
묶여 있어 웹·앱·백엔드가 같은 로직을 공유할 수 없었습니다. 셋째, "발표용 데모"와
"실제 서비스"가 코드 상에서 구분되지 않았습니다.

이번 1단계 리팩터링으로 첫째·둘째 문제를 해결했습니다(로직을 `core/`로 분리).
아래 계획은 그 위에서 실제 서비스로 나아가는 순서입니다.

## 데모 · 프로덕션 분리 전략

핵심은 **로직(core)은 하나로 공유하고, 그 위에 여러 "표현 계층"을 얹는 것**입니다.
표현 계층은 용도에 따라 나눕니다.

| 구분 | 목적 | 코드 위치(제안) | 상태 |
| --- | --- | --- | --- |
| 데모 계층 | 수업·발표에서 빠르게 보여주기 | `app.py`, `mobile_demo.py` | 유지 |
| 도메인 로직 | 예측·데이터 처리(공유 자산) | `core/` | 완료 |
| 서비스 API | 실제 서비스의 백엔드 | `backend/` (향후) | 예정 |
| 서비스 프런트 | 실제 사용자용 모바일 웹앱(PWA) | `frontend/` (향후) | 예정 |

이렇게 하면 발표용 데모는 지금처럼 Streamlit으로 가볍게 유지하면서, 실제 서비스는
`core/`를 공유하는 별도의 백엔드·프런트엔드로 독립적으로 키울 수 있습니다. 데모를
계속 고쳐도 서비스가 영향을 받지 않고, 그 반대도 마찬가지입니다.

## 목표 폴더 구조(향후)

지금 당장 모든 폴더를 만들 필요는 없습니다. 아래는 서비스로 확장했을 때의 목표
모습이며, 필요할 때 단계적으로 추가합니다. 현재 만들어 둔 부분은 `# 현재`로
표시했습니다.

```
seat-predictor/
├── core/                     # 공유 도메인 로직 (프레임워크 무관)   # 현재
│   ├── prediction.py
│   ├── seoul_api.py
│   └── cache.py
├── app.py                    # 데모: 웹 프로토타입                # 현재(유지)
├── mobile_demo.py            # 데모: 모바일 발표 데모              # 현재(유지)
├── seoul_api.py              # 하위호환 shim                      # 현재
├── backend/                  # 서비스 백엔드 API (FastAPI 등)      # 향후
│   ├── main.py               #   - core 를 import 해서 REST 로 노출
│   └── routers/
├── frontend/                 # 서비스 프런트 (PWA)                # 향후
│   ├── index.html
│   └── src/
├── scripts/                  # 검증·운영 스크립트                 # 현재
│   └── regression_check.py
├── docs/                     # 팀 문서                            # 현재
├── tests/                    # 자동화 테스트                      # 향후
├── requirements.txt
├── .streamlit/
│   └── secrets.toml.example  # 인증키 예시(실제 키는 커밋 금지)
└── README.md
```

## 배포로 가는 작업 순서(로드맵)

아래 순서는 "안전하게, 기존 기능을 깨지 않으면서" 확장하는 것을 우선합니다.

1단계 — 로직 분리(완료). 예측·데이터 로직을 `core/`로 옮기고 데모 두 개가 이를
공유하도록 정리했습니다. 회귀 검증으로 결과 동일을 확인했습니다.

2단계 — 테스트·품질 기반 다지기. `scripts/regression_check.py`를 `tests/`로
발전시키고, GitHub Actions 등에서 PR마다 자동 실행되게 합니다. 코드 스타일
도구(black, ruff)를 도입해 팀 협업 시 일관성을 유지합니다.

3단계 — 백엔드 API 분리. `core/`를 그대로 import 하는 얇은 FastAPI 서버를
`backend/`에 만듭니다. 예측 요청을 REST(JSON)로 받도록 하면, 이후 어떤 프런트든
(웹·PWA·네이티브 앱) 같은 API를 쓸 수 있습니다. 이때 서울 공공데이터 인증키는
서버 환경변수로 관리하고 프런트에는 절대 노출하지 않습니다.

4단계 — 실제 서비스 프런트(PWA) 구축. 자세한 기술 선택은
[`mobile-tech.md`](mobile-tech.md)를 참고하세요. 데모(`mobile_demo.py`)에서
검증한 화면 흐름을 그대로 PWA로 옮기고, 3단계 백엔드 API에 연결합니다.

5단계 — 배포·운영. 백엔드는 컨테이너(예: Docker)로 클라우드에, 프런트는 정적
호스팅(예: Vercel·Netlify·Cloudflare Pages)에 올립니다. 캐싱·에러 로깅·기본
모니터링을 붙여 실제 트래픽을 견디게 만듭니다.

## 보안·운영 주의사항

- 인증키(`.streamlit/secrets.toml`)는 절대 커밋하지 않습니다. 이미 `.gitignore`로
  제외되어 있습니다. 백엔드 단계에서는 환경변수나 시크릿 매니저로 옮깁니다.
- 서울 공공데이터는 응답이 크고(버스 하루 약 4만 건) 갱신 주기가 길므로, 캐싱을
  반드시 유지합니다. `core/cache.py`가 그 기반을 제공합니다.
- 좌석 확률은 추정치라는 안내 문구를 서비스 화면에서도 유지합니다. 실측이 아닌
  값을 실측처럼 보이게 하지 않는 것이 사용자 신뢰의 핵심입니다.
