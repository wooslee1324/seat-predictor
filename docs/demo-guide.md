# 수업 발표용 데모 가이드

수업 발표에서 **좌석 예측기**를 간단하고 명확하게 시연하기 위한 가이드입니다.
정식 출시 설명이 아니라, "무엇을·어떤 순서로 보여줄지"에 집중합니다.

핵심 메시지 한 줄: **"퇴근길에 지금 탈지, 몇 분 뒤에 탈지 — 앉아서 갈 확률로
알려주는 앱입니다."**

## 1. 발표 시연 순서 (약 3~5분)

1. **한 줄 소개** — 위 핵심 메시지로 시작합니다.
2. **모바일 데모 실행** — 스마트폰 화면처럼 생긴 `mobile_demo.py` 를 띄웁니다.
3. **추천 경로 1 시연(붐빔)** — 강남역 → 사당역, 18:30. 혼잡도와 "몇 분 뒤에 타면
   앉아서 갈 확률이 올라가는지"를 보여줍니다.
4. **추천 경로 2 시연(여유)** — 잠실역 → 강남역, 22:00. "지금 타도 좋다"는 반대
   상황을 보여줍니다.
5. **버스 모드 시연** — 버스로 바꿔 한 번 더 보여줍니다(아래 예시 참고).
6. **핵심 구분 설명** — "실시간 도착정보"와 "예상 좌석 확률"이 다르다는 점을
   한 문장으로 짚습니다(5번 항목 참고).
7. **마무리** — 데이터 출처(서울시 공공데이터)와 앞으로의 방향(실시간 연동·모바일
   웹앱)을 한 줄로 언급합니다.

> 팁: 발표 전에 앱을 미리 한 번 실행해 두면 로딩 없이 매끄럽게 진행됩니다.

## 2. 모바일 데모 실행 방법

프로젝트 루트에서 아래를 실행합니다.

```bash
# (처음 한 번) 가상환경 + 라이브러리
python3 -m venv .venv && source .venv/bin/activate
pip install -r requirements.txt

# 모바일 발표용 데모 실행
streamlit run mobile_demo.py
```

실행하면 브라우저에 `http://localhost:8501` 로 스마트폰 형태의 화면이 열립니다.
발표 때는 브라우저 창을 좁게 줄이면 실제 폰 화면처럼 보입니다.

기존 웹 대시보드(넓은 화면)를 보여주고 싶으면 `streamlit run app.py` 를 씁니다.
둘은 같은 예측 로직(`core/`)을 공유하므로 결과가 동일합니다.

## 3. (선택) 백엔드 실행과 통합 테스트

발표에는 **필요 없습니다.** 모바일 데모만으로 충분합니다. "실제 서비스로 확장 중"
이라는 점을 보여주고 싶을 때만 선택적으로 곁들입니다.

```bash
# (선택) FastAPI 백엔드 실행
pip install -r backend/requirements.txt
uvicorn backend.main:app --reload      # http://localhost:8000/docs 에서 API 문서 확인

# (선택) 통합 테스트 — 자세한 방법은 docs/backend-integration-test.md
python3 scripts/integration_test.py --station 강남역 --ars 23305
```

이 부분은 "발표 후 질문이 나오면 보여주는 보너스" 정도로 준비하면 됩니다.

## 4. 발표용 추천 경로 예시

| 상황 | 교통수단 | 출발 → 도착 | 시간 | 기대 화면 |
| --- | --- | --- | --- | --- |
| 붐빔(러시아워) | 지하철 | 강남역 → 사당역 | 18:30 | 높은 혼잡도 + "N분 뒤에 타면 확률↑" 추천 |
| 여유 | 지하철 | 잠실역 → 강남역 | 22:00 | 낮은 혼잡도 + "지금 타도 좋아요" |
| 버스 | 버스 | 강남역10번출구 → 사당역4번출구 | 18:30 | 버스 혼잡도 + 추천 대기 위치 |

두 가지 상반된 상황(붐빔 vs 여유)을 연달아 보여주면 앱의 핵심 가치가 잘 전달됩니다.

## 5. "실시간 도착정보" vs "예상 좌석 확률" 쉽게 설명

발표에서 이렇게 한 문장으로 구분하면 됩니다.

- **실시간 도착정보** = "다음 열차·버스가 **몇 분 뒤에 오는지**" — 서울시 API가
  주는 **사실**입니다.
- **예상 좌석 확률** = "그 차를 타면 **앉아서 갈 수 있을지**" — 혼잡도로 계산한
  **추정치**이지, 실제 좌석 수를 센 값이 아닙니다.

비유: **"버스는 3분 뒤 도착(사실). 근데 앉을 수 있을지는 붐빔 정도로 짐작(추정)."**

이 구분이 중요한 이유도 한 줄로: **"어떤 API도 '빈 좌석 개수'를 실시간으로 주지
않기 때문에, 우리는 혼잡도로 좌석 확률을 '예상'해서 보여줍니다."**

참고로 화면에서는 네 가지를 구분해 표기합니다: `실시간 도착정보` · `예상 혼잡도`
· `예상 좌석 확률` · `참고용 통계`.

## 6. API 키가 없어도 발표할 수 있습니다

인증키가 없어도 **앱은 정상적으로 돌아갑니다.** 자동으로 폴백(대체 동작)합니다.

- 역 목록: 실데이터를 못 받으면 **기본 역 목록**으로 표시.
- 혼잡도·좌석 확률: 실데이터가 없으면 **추정(Mock) 값**으로 계산해 그래프를 그림.
- 화면에는 실데이터인지(✅) 추정치인지(⚠️) **배지로 표시**되므로 정직하게 시연됩니다.

즉 **키 없이도 발표 흐름은 완전히 동일**합니다. 키가 있으면 지하철 혼잡도가
실데이터로 바뀌는 정도의 차이입니다. 발표 직전 키 설정에 실패해도 당황할 필요가
없습니다.
