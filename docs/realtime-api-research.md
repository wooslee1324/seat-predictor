# 서울시 실시간 API 조사 (좌석 예측 서비스용)

이 문서는 좌석 예측기를 실제 서비스로 발전시키기 위해, 서울시에서 쓸 수 있는
실시간 교통 API를 조사·정리한 것입니다. **실제 좌석 수를 직접 주는 API는 사실상
없기 때문에**, "실시간 교통 정보"를 입력으로 받아 "예상 좌석 확률"을 계산하는
구조를 전제로 합니다.

> ⚠️ 조사 기준일: 2026-07. 각 API의 정확한 필드명·요청 파라미터·제공 여부는
> 실제 신청 후 응답으로 반드시 재확인해야 합니다("확인 필요"로 표시).

## 0. 용어 정의 (화면·문서에서 항상 구분)

혼동을 막기 위해 아래 네 가지를 명확히 구분해서 표기합니다. 실제 API가 주지 않는
값을 "실시간"이라고 부르지 않는 것이 핵심 원칙입니다.

| 표기 | 뜻 | 출처 |
| --- | --- | --- |
| **실시간 도착 정보** | 지금 몇 분 뒤 열차/버스가 오는지 | 실시간 API가 직접 제공 |
| **예상 혼잡도** | 그 시각 칸/차량이 얼마나 붐빌지 | 실시간 혼잡도(있을 때) 또는 통계에서 추정 |
| **예상 좌석 확률** | 앉아서 갈 수 있을 확률 | 혼잡도로부터 유도한 **추정치**(실측 아님) |
| **참고용 통계** | 과거 평균 이용객/혼잡도 | 통계 API(과거·집계 데이터) |

## 1. 현재 프로젝트가 쓰는 API의 성격

현재 연동된 세 API는 **모두 통계/과거 데이터이며, 실시간이 아닙니다.**

| 서비스명 | 데이터 | 실시간성 | 현재 용도 |
| --- | --- | --- | --- |
| `subwConfusion` (지하철혼잡도정보) | 1~8호선 요일·시간대(30분 단위) **평균** 혼잡도 | ❌ 통계. 분기별 갱신 | 예측용 혼잡도(실시간 아님, 요일·시간대 평균) |
| `CardSubwayStatsNew` (지하철 승하차) | 역·노선별 **하루 총** 승하차 인원 | ❌ 통계. 약 3일 지연 | 참고용 통계(하루 이용객) |
| `CardBusStatisticsServiceNew` (버스 승하차) | 정류장·노선별 **하루 총** 승하차 인원 | ❌ 통계. 약 3일 지연 | 참고용 통계(하루 이용객) |

즉 현재 "혼잡도"는 엄밀히는 **"해당 요일·시간대의 과거 평균 혼잡도"**입니다. 지금
이 순간의 실제 붐빔이 아니므로, 화면에서 "실시간"이라는 표현은 쓰지 않고 "예상
혼잡도(시간대 평균 기반)"로 표기하는 것이 맞습니다.

## 2. 서울시에서 쓸 수 있는 실시간 API 후보

### 2-1. 실시간 지하철 도착 정보 (`realtimeStationArrival`)

- **제공 데이터**: 특정 역 기준, 각 호선·방향별로 다음 열차의 도착 예정 메시지
  ("전역 출발", "2분 후 도착" 등), 도착 예정 시간, 열차 종류(급행/일반), 종착역,
  상하행 구분, 데이터 생성 시각(`recptnDt`).
- **실시간성**: ✅ 실시간. 단, `recptnDt`(생성 시각)와 현재 시각의 차이만큼 열차가
  더 진행한 것으로 보정해서 써야 함.
- **인증키**: 필요(서울 열린데이터광장에서 발급).
- **좌석 예측 활용**: 열차가 "몇 분 뒤 오는지"를 실제 값으로 제공 → 지금 우리가
  mock으로 만드는 "N분 뒤 탑승" 시나리오를 실데이터로 대체할 수 있음. 다만 이
  API 자체는 **혼잡도/좌석 정보를 주지 않음**.
- **한계**: 서울 구간만 제공. 칸별 혼잡도·좌석 정보 없음. 도착 시간은 예측값이라
  실제와 오차 가능.

### 2-2. 실시간 지하철 열차 위치 정보 (`realtimePosition`)

- **제공 데이터**: 호선별로 현재 운행 중인 각 열차가 어느 역 구간에 있는지(위치),
  상하행, 급행 여부, 열차 번호 등.
- **실시간성**: ✅ 실시간(생성 시각 보정 필요).
- **인증키**: 필요(열린데이터광장).
- **좌석 예측 활용**: "다음 열차가 지금 어디쯤 있는지"로 도착 시각을 더 정밀하게
  보정하거나, 열차별로 예측을 매칭하는 데 활용 가능.
- **한계**: 위치만 제공, **혼잡도·좌석 정보 없음**. 서울 구간 한정.

### 2-3. 실시간 버스 도착 정보 (버스도착정보조회)

- **제공 데이터**: 정류소·노선 기준, 첫째·둘째 도착 예정 버스의 도착 예정 시간(초),
  차량번호, 버스 유형, **혼잡도(붐빔 단계)**, **만차 여부**, 막차 여부 등.
- **실시간성**: ✅ 실시간.
- **인증키**: 필요(TOPIS / 공공데이터포털 `data.go.kr`, 서울 열린데이터광장에도
  유사 목록 있음).
- **좌석 예측 활용**: **버스 모드에서 가장 유용.** 도착 시간 + 실시간 혼잡도 단계를
  함께 제공하므로, 통계 대신 **실시간 혼잡도 → 예상 좌석 확률** 계산이 가능.
- **한계**: 혼잡도는 보통 몇 단계의 범주값(여유/보통/혼잡 등)이라 정밀 좌석 수는
  아님. **좌석버스 잔여좌석수(`remainSeatCnt`류)는 광역·좌석버스 위주로만 제공되고,
  일반 시내버스는 미제공(-1)인 경우가 많음** — 확인 필요. 노선ID·정류소ID를 먼저
  알아야 조회 가능.

### 2-4. 실시간 버스 위치 정보 (버스위치정보조회)

- **제공 데이터**: 노선 단위로 현재 운행 중인 버스들의 위치(정류소 구간), 정류소
  도착 여부, 차량번호·유형, **혼잡도**, 일부 재차 인원(확인 필요).
- **실시간성**: ✅ 실시간.
- **인증키**: 필요(공공데이터포털/TOPIS).
- **좌석 예측 활용**: 특정 노선 버스가 지금 어디에 있는지 + 혼잡도로 "곧 올 버스가
  붐비는지"를 판단 → 예상 좌석 확률 보정.
- **한계**: 노선 기준 조회라 "내 정류소 관점" 가공이 필요. 혼잡도는 범주값.

### 2-5. (보조) 실시간 칸별 혼잡도 — 서울 열린데이터광장 **밖**

- **제공 데이터**: 진입 역 기준, 열차/칸별 실시간 혼잡도(예: TMAP 대중교통 API는
  10분 간격 칸별 혼잡도 제공, SK 지오비전 퍼즐 등).
- **실시간성**: ✅ 준실시간(10분 간격 등).
- **인증키**: 필요. **서울 공공데이터가 아니라 민간(SK/TMAP) API**라 별도 신청·약관·
  호출 한도·비용 정책 확인 필요.
- **좌석 예측 활용**: 지하철 좌석 예측의 정확도를 가장 크게 높일 수 있는 데이터
  (칸 단위 실시간 혼잡). "추천 대기 위치(몇-몇번 문)"를 실제 칸 혼잡도로 뒷받침 가능.
- **한계**: 서울 열린데이터광장 소속이 아님(외부 의존성·비용 리스크). 커버리지·정확도
  사전 검증 필요.

### 2-6. 운행 지연 / 사고 정보

- **현황**: 서울 열린데이터광장에 "지연·사고"만 주는 깔끔한 단일 실시간 API는
  뚜렷하지 않음(확인 필요). 실무에서는 (1) 실시간 도착정보 메시지의 지연 문구,
  (2) 서울교통공사 공지/알림, (3) 별도 뉴스·알림 채널로 간접 파악하는 경우가 많음.
- **좌석 예측 활용**: 지연이 감지되면 "예측 신뢰도 낮음" 배지를 띄우거나 예측을
  보류하는 안전장치로 활용.
- **한계**: 표준화된 실시간 사고 API 부재 → 초기에는 도착정보 메시지 파싱 수준으로만
  접근 권장.

## 3. 핵심 결론 — "실시간 교통 정보 + 예상 좌석 확률" 설계

**어떤 API도 "이 열차/버스에 지금 빈 좌석이 몇 개"를 정확히 주지 않습니다.**
버스 혼잡도(범주값)·칸 혼잡도(외부 API)가 가장 근접한 실시간 신호입니다. 따라서
서비스는 다음 3계층으로 명확히 분리해 설계합니다.

```
[실시간 입력 계층]                     [예측 계층]              [출력 계층]
- 실시간 도착정보(지하철/버스)   ─┐
- 실시간 혼잡도(버스/칸 혼잡도)  ─┼─▶  예상 좌석 확률 계산  ─▶  · 실시간 도착 정보(사실)
- 실시간 위치정보                ─┘     (core.prediction)        · 예상 혼잡도(추정)
                                                                 · 예상 좌석 확률(추정)
[통계 입력 계층]                                                 · 참고용 통계(과거)
- 지하철혼잡도(시간대 평균)      ───▶  (실시간 없을 때 폴백)      + 데이터 출처 배지
- 승하차 통계                    ───▶  참고용 통계로만 표시
```

### 데이터 소스 우선순위(폴백 사다리)

좌석 확률 계산에 쓰는 "혼잡도"는 다음 순서로 가장 실시간에 가까운 값을 채택하고,
**화면에는 어떤 소스를 썼는지 배지로 반드시 표시**합니다.

1. 실시간 칸/차량 혼잡도(버스 도착정보 혼잡도, 또는 외부 칸 혼잡도) → `실시간`
2. 없으면 시간대 평균 혼잡도(`subwConfusion`) → `예상(시간대 평균)`
3. 그것도 없으면 러시아워 곡선 mock → `추정(Mock)`

지금 백엔드 `POST /predict` 응답의 `data_source`(실데이터/추정)를 이 3단계로
확장하면 됩니다.

### 화면/문구 규칙

- 열차·버스가 오는 시각은 실시간 API가 줄 때만 "실시간 도착 정보"로 표기.
- 좌석은 언제나 "예상 좌석 확률(추정)"으로 표기하고 실측이라고 하지 않음.
- 통계에서 온 값은 "참고용 통계(과거 평균)"로 표기.
- 실시간 혼잡도가 없을 때는 "예상 혼잡도(시간대 평균 기반)"로 표기.

## 4. 향후 백엔드 endpoint 초안 (제안)

현재 `backend/`의 `/health`·`/stations`·`/predict` 위에 아래를 단계적으로 추가하는
것을 제안합니다. 실시간 계층은 외부 API 의존·호출 한도가 있으므로 **짧은 캐시
(수십 초)**를 두고, 실패 시 통계/mock으로 폴백합니다.

- `GET /realtime/subway/arrivals?station={역명}`
  → 실시간 지하철 도착 정보(다음 열차 도착 예정). 출처: `realtimeStationArrival`.
- `GET /realtime/subway/positions?line={호선}` (선택)
  → 실시간 열차 위치. 출처: `realtimePosition`.
- `GET /realtime/bus/arrivals?stopId={정류소}&routeId={노선}`
  → 실시간 버스 도착 예정 + 실시간 혼잡도/만차 여부. 출처: 버스도착정보조회.
- `POST /predict` 확장
  → 요청에 `use_realtime: bool` 추가. 응답 `data_source`를
    `실시간 | 예상(시간대 평균) | 추정(Mock)` 3값으로 확장하고,
    `realtime_arrivals`(있으면) 필드를 함께 반환.
- (선택) `GET /realtime/status`
  → 운행 지연/사고 요약(초기에는 도착정보 메시지 기반 간이 판정).

각 실시간 endpoint는 응답에 **데이터 소스와 생성 시각(및 보정 여부)**을 함께 담아,
프런트가 "실시간/예상/통계"를 구분해 표시할 수 있게 합니다.

## 5. 인증키·운영 주의사항

- 실시간 API는 서울 열린데이터광장과 공공데이터포털(`data.go.kr`)에 나뉘어 있고,
  버스 계열은 TOPIS/공공데이터포털 발급 키가 필요한 경우가 많음 → **어느 포털에서
  어떤 키를 발급받아야 하는지 API별로 확인 필요.**
- 인증키 파일(`.streamlit/secrets.toml` 등)은 **절대 커밋 금지**(이미 `.gitignore`
  제외). 백엔드는 환경변수로 키를 주입.
- 실시간 API는 호출 한도가 있으므로 정류소/역 단위로 **짧은 캐시**를 두고, 통계
  데이터(분기·일 갱신)와는 캐시 정책을 분리.
- 외부(SK/TMAP) 칸 혼잡도는 비용·약관 리스크가 있으므로 도입 전 검증.
